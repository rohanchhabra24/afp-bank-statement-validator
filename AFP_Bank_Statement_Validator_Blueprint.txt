1.  Problem Statement & Reality Check
1.1  What AFP Actually Is
AFP (Advanced Function Presentation) is IBM's binary page-description language — the financial industry's PostScript. Banks use it because it has been in production since the 1980s, it is exact (pixel-perfect), and it is deeply embedded in mainframe print pipelines. The format stores structured objects: pages, data objects, presentation text, graphics, and overlays, all encoded in a proprietary binary stream with structured fields (SFs) and tagged logical elements (TLEs).

⚠  Hard Truth About AFP Parsing
  › There is no lightweight open-source AFP parser in Python or Node that handles complex bank statements out of the box
  › Apache FOP generates AFP; it does not consume it — people confuse these
  › IBM's own AFP tooling (AFP Workbench, AFP Viewer) is Windows-only, expensive, and has no API
  › afplib (Python) is the closest open-source option — partial support, handles basic MO:DCA structures
  › Realistic POC path: AFP → PDF via a conversion step first, then parse the PDF

Hackathon Recommendation

  › If you do NOT have real AFP yet: build the full validation + highlight pipeline on PDF/CSV, plug AFP conversion in later
  › Either way — present the architecture as complete; the conversion step is a connector, not the core value

Step 2 — Table Structure Extraction
Bank statements from AFP have a fixed columnar layout. We exploit this by treating the document as a coordinate grid:
	•	Extract all text tokens with their (x, y, page) bounding boxes
	•	Cluster tokens into rows by y-coordinate proximity (within ±3pt = same row)
	•	Cluster columns by x-coordinate ranges — since AFP is pixel-exact, column positions are consistent across all rows
	•	Map columns to semantic roles: Party Name, Debit, Credit, Balance using header row labels OR positional heuristics (rightmost numeric columns = financial columns)
	•	Output: a DataFrame / JSON array of structured rows — each row is one transaction

Step 3 — Validation Engine
Each validation rule is an independent module that takes a row (or a set of rows) and returns a list of ValidationIssue objects. This makes the rules easy to demo, toggle, and extend.

Step 4 — Annotated Output Generation
The output is a PDF where each validation issue is visually marked. We use reportlab (Python) or pdfplumber + PyMuPDF to overlay highlights and margin annotations:
	•	RED highlight — critical errors (impossible date, balance mismatch, column bleed)
	•	AMBER highlight — warnings (format inconsistency, locale ambiguity)
	•	GREEN annotation — auto-corrected values shown in margin
	•	Summary table on last page — all issues with severity, row reference, rule name
	•	Machine-readable sidecar JSON — same issue list for downstream systems

3.3  Data Flow — Detailed

Request Lifecycle
  › 1.  Client POST /validate  →  uploads AFP file(s)
  › 2.  AFP Converter decodes MO:DCA structure → Page objects with positioned text tokens
  › 3.  Table Extractor groups tokens into rows (y-clustering) and columns (x-range mapping)
  › 4.  Schema Detector identifies column semantics: DATE | PARTY | DEBIT | CREDIT | BALANCE
  › 5.  Validation Runner passes each row through each active rule module
  › 6.  Issue Aggregator deduplicates and scores issues (CRITICAL / WARNING / INFO)
  › 7.  Annotator overlays coloured highlights on source PDF at original bounding-box coordinates
  › 8.  Response: { annotated_pdf: base64, issues: [...], summary: {...} }

4.3  Production-Grade Improvements

	•	Message queue (Kafka / RabbitMQ): decouple file ingestion from processing — handle burst loads from banks submitting end-of-day batches
	•	Distributed processing: each AFP file processed by a worker pod (Kubernetes) — horizontally scalable
	•	Result caching: SHA-256 hash of input AFP → skip reprocessing unchanged files
	•	Webhook callbacks: bank systems receive a callback when their document is processed — no polling
	•	Role-based output: compliance officer sees full annotations; bank clerk sees only correctable errors; API consumer sees raw JSON
	•	SOC 2 / data residency: PII masking option — hash party names in transit, unmask only in authorised rendering layer
	•	Multi-format input: extend to handle SWIFT MT statements, CSV exports, OFX — same validation engine, different parsers

5.3  Risk Mitigation

What Can Go Wrong & Our Fallbacks
  › AFP parsing fails on real files → fallback: pre-convert 3–5 representative files to PDF, demo on those, frame AFP conversion as 'solved via IBM tooling in production'
  › Column detection is off for this bank's layout → fallback: hardcode column x-ranges for demo file, show the configurable YAML schema for others
  › PyMuPDF coordinate mapping is off by 1–2px → fallback: paragraph-level highlights instead of word-level — still looks great
  › Frontend not ready → fallback: Swagger UI + Postman demo is fine for technical judges
  › Balance reconciliation too slow on large file → fallback: process only first 100 rows in demo, note O(n) complexity is fine for production scale
